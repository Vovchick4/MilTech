# Урок №111. Стек і Купа

Пам'ять програми поділяється на дві основні області: стек (stack) і купа (heap).

## Стек (Stack)

- Автоматичне керування
- Дуже швидкий
- Обмежений розмір (~1-8 МБ)
- Локальні змінні, параметри функцій

```c++
void func()
{
    int x = 5;       // стек: автоматично
    int arr[100];    // стек
}                    // автоматично звільняється
```

## Купа (Heap)

- Ручне керування (new/delete)
- Повільніший
- Великий розмір (доступна RAM)
- Динамічні об'єкти

```c++
void func()
{
    int* p = new int(5);   // купа
    int* arr = new int[1000];  // купа
    delete p;              // ручне звільнення
    delete[] arr;
}
```

## Візуалізація

```
Стек (згори вниз):
┌──────────────────┐
│   func()         │ ← поточна функція
│   main()         │
│   ...            │
└──────────────────┘

Купа (знизу вгору):
┌──────────────────┐
│   динамічні дані │
│   new / malloc   │
└──────────────────┘
```

## Порівняння

| Характеристика | Стек | Купа |
|---------------|------|------|
| Швидкість | Дуже швидко | Повільно |
| Розмір | Малий (1-8 МБ) | Великий (RAM) |
| Керування | Автоматичне | Ручне |
| Фрагментація | Немає | Можлива |
| Доступ | LIFO | Довільний |

## Чому стек обмежений?

Кожен потік має свій стек. Великі об'єкти на стеку → переповнення стеку (stack overflow):

```c++
void bad()
{
    int huge[1000000];  // stack overflow! (4 МБ)
}

void good()
{
    int* huge = new int[1000000];  // купа — OK
    delete[] huge;
}
```

## Вибір: стек vs купа

```c++
// Стек — малі об'єкти, відомий розмір
int x = 5;
std::array<int, 100> localArr;

// Купа — великі об'єкти, динамічний розмір
std::vector<int> dynamicArr(1000000);
int* bigData = new int[1000000];
```

## Кращі практики

```c++
// Сучасний C++:
std::vector<int> v(1000);       // купа, але автоматичне керування
auto ptr = std::make_unique<int>(5);  // купа, автоматичне звільнення
```

## Правило

Використовуйте стек для малих об'єктів із коротким часом життя. Використовуйте купу для великих даних або об'єктів, що мають "жити" довше функції.
